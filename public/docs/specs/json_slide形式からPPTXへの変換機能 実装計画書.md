# **json\_slide形式からPPTXへの変換機能 実装計画書**

## **1\. 背景と目的**

本プロジェクトは、LLMによって生成された構造化データ（json\_slide）およびそれをレンダリングしたReactコンポーネント群を、クライアントサイドで直接.pptx形式に変換・エクスポートする機能を実装するものです。

過去の検証用スクリプト（pptxGenerator.js）は破棄し、「Genspark風AIスライド生成メカニズム調査」の知見に基づき、**PptxGenJS**を中核としたゼロベースでの新規実装を行います。

最大の目標は\*\*「ブラウザ上の見た目を極力維持しつつ、編集可能な標準PPTXオブジェクトと、高精度な画像スナップショットを最適に組み合わせる」\*\*ことです。アニメーションは対象外とします。

## **2\. アーキテクチャ概要**

変換処理は以下の3つの主要レイヤーで構成されます。

### **Layer 1: JSON to PPTX Translation (翻訳層)**

json\_slide のデータ構造を解析し、スライドのタイプ（TitleSlide, ContentSlide, ChartSlide等）に応じて、PptxGenJSの対応する描画メソッド（addText, addShape, addTable）を呼び出します。

### **Layer 2: Theme Engine (テーマエンジン)**

React側で適用されているテーマのCSS変数やフォント設定を解析し、PptxGenJSのマスター機能（defineSlideMaster）にマッピングします。これにより、背景色や一貫したフォントスタイルをPPTX全体に適用します。

### **Layer 3: Snapshot Layer (スナップショット層)**

チャート（Recharts等）、複雑なSVG、CSSグラデーションを多用したプロセスフローなど、PPTXのネイティブAPIで再現困難またはコストが高いコンポーネントを処理します。

バックグラウンドで対象要素を html-to-image や html2canvas を用いてDataURL（PNG）に変換し、PptxGenJSの addImage としてスライドに埋め込みます。

## **3\. レンダリング・ハイブリッド戦略**

すべてを画像化するのではなく、テキストの編集可能性を残すための「ハイブリッド・アプローチ」を採用します。

| 要素 / スライドタイプ | レンダリング方式 | 詳細・対応方針 |
| :---- | :---- | :---- |
| **タイトル・見出し** | **ネイティブ (Text)** | 編集性を維持。フォントサイズ・色・位置を計算して配置。 |
| **本文・リスト** | **ネイティブ (Text)** | 箇条書き（bullet）設定や行間設定をPptxGenJSのプロパティで再現。 |
| **テーブル (表)** | **ネイティブ (Table)** | addTableを使用。背景色やボーダーを再現し、データ編集を可能に。 |
| **シンプルな図形** | **ネイティブ (Shape)** | 区切り線、背景のアクセントブロック等は addShape で描画。 |
| **チャート (Chart)** | **スナップショット (Image)** | 再現性を最優先し、React側のレンダリング結果を画像化して配置。 |
| **フロー図・KPI** | **スナップショット (Image)** | 複雑なレイアウト崩れを防ぐため、コンポーネント単位で画像化。 |

## **4\. 処理フロー (Export Pipeline)**

PPTXエクスポートボタンが押下されてからの内部処理フローです。

1. **Pre-processing (前処理):**  
   * 選択されているテーマ情報を取得し、対象テーマが modern-indigo であるか検証（対象外の場合はエラーを表示し処理を中断）。  
   * PptxGenJSインスタンスを初期化。  
   * defineSlideMaster でマスターレイアウト（背景色、共通ヘッダー/フッター）を登録。  
2. **Snapshot Extraction (非同期画像化フェーズ):**  
   * 現在のDOMから、画像化が必要な対象（チャート等）を特定。  
   * html-to-image を用いて、対象のDOM要素をPNG画像（Base64）に変換。すべての変換が完了するまでPromiseで待機。  
   * *※非表示状態のタブ/スライドにある要素を描画するため、一時的なオフスクリーンDOMを利用する工夫が必要。*  
3. **Slide Generation (スライド生成フェーズ):**  
   * json\_slide.slides の配列をループ処理。  
   * スライドごとの type に応じたマッパー関数（例: renderTitleSlide(slideData, pptx)）を実行。  
   * テキストや図形を配置し、事前取得したBase64画像を適切な座標・サイズで挿入。  
4. **File Save (ファイル保存):**  
   * pptx.writeFile({ fileName: "Presentation.pptx" }) を実行し、ユーザーにダウンロードを提供。

## **5\. 実装のステップ**

ゼロから開発を進めるための推奨ステップです。

* **Step 1: 基盤構築とマスター定義**  
  * 新規ファイル src/utils/pptxExportEngine.ts (または .js) を作成。  
  * PptxGenJSの基本初期化、modern-indigoに対応するマスターレイアウト定義の実装。対象外テーマでのエクスポートブロック処理を追加。  
* **Step 2: テキストベースのスライド対応**  
  * TitleSlide, ContentSlide, AgendaSlide など、ネイティブテキストと基本図形だけで構成できるコンポーネントの変換ロジックを実装。  
* **Step 3: スナップショット・エンジンの統合**  
  * html-to-image パッケージの導入。  
  * ChartSlide や ProcessFlowSlide を対象に、特定IDのDOMをキャプチャし、PPTXに画像として貼り付けるロジックの構築と非同期制御の実装。  
* **Step 4: 高度なネイティブ要素（Table等）の対応**  
  * TableSlide などのデータをPptxGenJSのテーブルオブジェクトにパースし、罫線やセル背景色をマッピングする処理の追加。  
* **Step 5: 調整（対象テーマの洗練）**  
  * エクスポートされたPPTXと実際のReact画面を見比べ、座標・フォントサイズ（px to pt変換係数）の微調整。今回は modern-indigo の再現性向上に注力。

## **6\. 技術的な課題と対策案**

1. **DOMの解像度（Retina対応）**  
   * **課題:** 標準の html-to-image では画像がぼやける場合がある。  
   * **対策:** キャプチャ時に pixelRatio: 2 などを指定し、高解像度でレンダリングした後にPPTX上で縮小配置する。  
2. **非表示スライドのキャプチャ問題**  
   * **課題:** Reactの画面上で現在アクティブでないスライド（または仮想スクロールで破棄されたDOM）はキャプチャできない。  
   * **対策:** エクスポート実行時、画面外（または z-index: \-9999）の隠しコンテナに全ての「スナップショット対象スライド」を一括レンダリングし、マウント完了後にキャプチャ処理を走らせる。  
3. **CSS単位（px/rem）とPPTX単位（inches/points）の変換**  
   * **課題:** 見た目のレイアウト比率を一致させるための計算が必要。  
   * **対策:** 基本的なスライドサイズ（16:9 \= 10インチ x 5.625インチ）を定義し、React側の基準幅（例: 960px）に対するスケールファクター（定数）を設けて座標を一括変換するユーティリティ関数を用意する。
