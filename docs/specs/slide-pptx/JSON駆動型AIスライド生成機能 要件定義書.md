# **JSON駆動型AIスライド生成機能 要件定義書**

## **1\. プロジェクト概要**

### **1.1 背景と目的**

現在のシステムでは、LLMにHTMLを直接出力させる html\_slide タイプのArtifact機能が存在するが、タグの破損やレイアウト崩れ、デザインの柔軟性の欠如といった課題がある。

本プロジェクトでは、Gensparkのアーキテクチャに倣い、LLMには構造化されたJSONデータのみを出力させ、フロントエンド側のReactコンポーネントで動的にスライドを描画する全く新しいスライド生成機能を実装する。これにより、堅牢でリッチなUI/UXを持つプレゼンテーション表示を実現する。

### **1.2 スコープ**

* **対象（In-Scope）**  
  * 新規Artifactタイプ（例: json\_slide）の定義と、フロントエンドでのパース処理の拡張。  
  * LLMが出力すべきスライド用JSONスキーマの策定。  
  * 既存の ArtifactPanel とは独立した、新しいスライド専用パネル（例: PresentationPanel）の新設。  
  * JSONデータに基づき、動的にスライドをレンダリングするReactコンポーネント群の実装。  
  * Tailwind CSSを用いたテーマ適用と、Framer Motionを用いたページ遷移アニメーションの実装。  
  * LLMの出力ブレに起因するレイアウト崩壊を防ぐためのフロントエンド安全対策の実装。  
* **対象外（Out-of-Scope）**  
  * ドラッグ＆ドロップ、テキストの直接編集、レイアウト変更などの手動編集機能。  
  * PDFやPPTX形式へのエクスポート・ダウンロード機能。  
  * Mermaid.js等の動的図解描画、画像生成AIとの非同期連携（今回はテキストベースの静的なレイアウト確立を優先）。  
  * 既存の html\_slide 機能の削除や改修（既存ルートはそのまま維持する）。

## **2\. アーキテクチャ・データフロー要件**

### **2.1 新規Artifactルートの確立**

既存の仕組みを壊さないよう、Difyのレスポンスから受け取る artifact\_type に新しい値を定義する。

* **既存**: artifact\_type: "html\_slide" \-\> 既存の ArtifactPanel でHTMLとして描画。  
* **新規**: artifact\_type: "json\_slide" \-\> 新設する PresentationPanel にルーティングし、JSONデータとして描画。

### **2.2 パース処理の拡張**

* フロントエンドの utils/responseParser.ts（または該当のArtifact処理部）を拡張する。  
* artifact\_type が json\_slide の場合、artifact\_content （文字列で送られてくるJSON）を JSON.parse() でJavaScriptオブジェクトに変換し、スライド用コンポーネントへPropsとして引き渡す。

## **3\. データモデル（JSONスキーマ）要件**

LLMが生成し、フロントエンドが受け取るJSONデータの構造を以下のように定義する。

{  
  "presentation\_title": "AIの未来とビジネス活用",  
  "theme": "corporate-modern", // デザイントークン（例: corporate-modern, creative-dark, minimal-light）  
  "slides": \[  
    {  
      "id": "slide\_1",  
      "layout\_type": "title\_slide", // title\_slide, content\_slide, split\_slide 等  
      "content": {  
        "title": "AIの未来とビジネス活用",  
        "subtitle": "Generative AIがもたらすパラダイムシフト",  
        "author": "AI推進プロジェクトチーム"  
      }  
    },  
    {  
      "id": "slide\_2",  
      "layout\_type": "split\_slide",  
      "content": {  
        "title": "現在の課題とAIによる解決",  
        "left\_column": \[  
          "属人的な業務プロセスの限界",  
          "膨大なデータ処理コストの増大"  
        \],  
        "right\_column": \[  
          "自動化によるプロセス標準化",  
          "LLMを用いた非構造化データの瞬時分析"  
        \]  
      }  
    }  
  \]  
}

### **3.1 レイアウトタイプ (layout\_type) の種類**

初期フェーズとして、以下の代表的なスライドレイアウトをサポートする。

1. title\_slide: タイトル、サブタイトル、発表者名などを中央配置する表紙スライド。  
2. content\_slide: タイトルと、1カラムのテキストまたは箇条書きのリストを持つ汎用スライド。  
3. split\_slide: タイトルと、左右2カラムに分割されたコンテンツを持つ比較・対比スライド。  
4. quote\_slide: 引用文や重要なメッセージを強調して全画面で表示するスライド。

## **4\. UI/UX・フロントエンド実装要件**

### **4.1 専用パネル（PresentationPanel）の新設**

既存のチャットUIや ArtifactPanel を汚染しないよう、スライド表示に特化した専用コンポーネントを作成する。

* **構成要素**:  
  * **ビューポート**: アスペクト比 16:9 で固定されたスライド表示領域。コンテナサイズに合わせてレスポンシブにスケールダウン/アップする。  
  * **ナビゲーションバー**: スライドの下部（または上部）に配置。前へ/次へのボタン、現在のスライド番号/総スライド数（例: 2 / 10）を表示。  
* **状態管理**:  
  * コンポーネント内で currentSlideIndex を保持し、表示するスライドを切り替える。

### **4.2 Reactコンポーネントによるレンダリング**

slides 配列をマッピングし、layout\_type に応じて適切なコンポーネントを動的にマウントする。

* \<SlideRenderer slide={slideData} theme={themeToken} /\> のような親コンポーネントから、\<TitleSlide /\>, \<SplitSlide /\> 等に分岐させる設計とする。

### **4.3 スタイリングとテーマ管理（Tailwind CSS）**

* LLMが指定した theme トークンに基づき、最上位コンテナ（または各スライドコンテナ）に特定のCSSクラス（例: theme-corporate-modern）を付与する。  
* Tailwind CSSのクラスを用いて、テーマごとの背景色、テキスト色、フォントファミリーを適用する。

### **4.4 アニメーション（Framer Motion）**

* **ページ遷移**: スライドが切り替わる際、フェードインやスライドイン等のスムーズなトランジションをFramer Motionを用いて実装する。  
* **要素表示**: 各スライド内の要素（タイトル、リスト項目等）に対して、わずかな時間差（stagger）を持たせたフェードインアニメーションを付与する。

### **4.5 レイアウト安全対策（文字数ブレへの対応）**

LLMが生成するテキスト量のブレによってレイアウトが崩壊することを防ぐため、以下の対策を講じる。

1. **柔軟なコンポーネント配置**:  
   絶対配置を避け、FlexboxやCSS Gridを用いた相対レイアウトを徹底する。余白を適切に消費する（flex-grow等）ことで要素同士の重なりを防止する。  
2. **オーバーフロー制御と省略表示**:  
   タイトルや見出し等、高さを固定すべき要素には line-clamp クラスを適用し、想定文字数を超過した場合は三点リーダー（...）で省略表示にする。  
3. **コンポーネントのサイズ自動調整**:  
   コンテナクエリ（@container）等を利用し、要素内の文字量に応じてフォントサイズやパディングを段階的に縮小し、見切れを防ぐ。フェイルセーフとして親コンテナにスクロール（overflow-y-auto）を許容する設定を検討する。  
4. **データのフォールバック（オプショナルチェーン）**:  
   データ欠損（content.titleがない等）によるレンダリングエラーを防ぐため、コンポーネント側でデフォルト値設定やオプショナルチェーン（?.）を徹底する。配列要素に対しても型チェックを行い白画面エラーを回避する。  
5. **（運用面）LLMプロンプトの制約**:  
   システムプロンプト側で「タイトルは30文字以内」「箇条書きは5項目まで」等の明確な制約を設け、フロントエンドの許容量を超えるデータを抑止する。

## **5\. 非機能要件**

* **耐障害性（エラーハンドリング）**:  
  * LLMから返却されたJSONがパース不可能な場合（構文エラー等）、画面をクラッシュさせず、ユーザーに対して「スライドデータの生成に失敗しました」というフォールバックUIを表示する。  
  * 指定された layout\_type に対応するコンポーネントが存在しない場合は、デフォルトの content\_slide としてフォールバック描画するか、エラーをログに出力してスキップする。  
* **パフォーマンス**:  
  * 過度な再レンダリングを防ぐため、スライドデータが更新された時、またはページ送りの時のみ適切にコンポーネントが更新されるよう最適化を検討する。
