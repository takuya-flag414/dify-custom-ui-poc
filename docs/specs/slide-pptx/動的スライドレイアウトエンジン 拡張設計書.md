# **動的スライドレイアウトエンジン 拡張設計書**

## **テーマ: 視線誘導、ネスト構造、および高次ブロック戦略**

本ドキュメントは、「動的スライドレイアウトエンジン 基本設計書」に対する高度な拡張仕様を定義する。

LLMの「デザインの自由度（CSS等の直接操作）」を制限しつつも、「情報の強調意図（視線誘導）」、「複雑な階層表現（ネスト）」、および「高度なビジネス図解・ナビゲーション」を完全に担保するためのアーキテクチャである。

## **1\. セマンティック視線誘導 (Semantic Direction & Emphasis)**

LLMが「どう見せたいか（スタイル）」ではなく、「何を見せたいか（意味・意図）」を出力し、それをUIのテーマエンジンが最適な装飾に翻訳するシステム。

### **1.1 視覚的強調（Emphasis / Highlight）**

要素をハイライトさせ、ユーザーの視線を釘付けにするためのプロパティ。

* **JSONスキーマ定義:**  
  各ブロック、またはブロック内の特定の要素に対して emphasis プロパティ（列挙型）を付与できる。  
  {  
    "type": "comparison\_table",  
    "emphasis\_row": 1, // データの2行目を強調  
    "rows": \[  
      \["A案", "低コスト", "時間がかかる"\],  
      \["B案", "高コスト", "即効性がある"\] // この行が強調される  
    \]  
  }

* **テーマごとの翻訳ロジック（React描画時の処理）:**  
  * theme-corporate-modern（コンサル風）の場合: 強調行の背景色を \#EBF4FA (薄い青)にし、テキストを太字にする。  
  * theme-tech-startup（ベンチャー風）の場合: 強調行の境界線に 2px solid var(--accent-neon) を付与し、Z-indexを上げて浮き出させる（Drop shadow）。

### **1.2 視線フローの制御（Direction）**

プロセスやタイムラインにおいて、ユーザーの視線をどのように誘導するかを決定するプロパティ。

* **JSONスキーマ定義:**  
  {  
    "type": "process\_flow",  
    "direction": "horizontal", // "horizontal" (左から右) または "vertical" (上から下)  
    "show\_arrows": true,  
    "steps": \["現状分析", "課題特定", "施策立案"\]  
  }

* **UI処理とGrid配置への影響:**  
  * direction: "horizontal" の場合、内部の要素間にテーマに沿った「➔」アイコンや結合線をレンダリングする。  
  * 推論エンジンはこの direction を読み取り、もし horizontal で要素数が多い場合は、横幅を広く取れる RowStackGrid の下段に自動的に配置する。

## **2\. コンテナ・ブロックとネスト構造 (Container & Nested Blocks)**

複雑なビジネス資料（例：カードの中にグラフを配置する、左右分割の片方にさらにリストを配置する）を表現するため、「自身の中に blocks 配列を持てるブロック（コンテナ）」を導入する。

### **2.1 コンテナ・ブロックのスキーマ定義**

特定の type（例: card, column\_group）は、再帰的に blocks プロパティを持つことができる。

{  
  "type": "card",  
  "title": "A社のパフォーマンス",  
  "emphasis": "high",  
  "blocks": \[  
    {  
      "type": "text",  
      "content": "売上高は前年比150%を達成し、業界トップの成長率を記録。"  
    },  
    {  
      "type": "chart",  
      "chart\_type": "bar",  
      "data": \[/\* ... \*/\]  
    }  
  \]  
}

### **2.2 推論エンジンにおける「再帰的重み計算」**

ネストを許容した場合、推論エンジンは「親コンテナの重み」を正確に計算しなければならない。

* **基本ルール**: コンテナの重み \= コンテナ自身の基本重み \+ 内部(blocks)の重みの合計。  
* **計算例**: 上記JSONの card の場合、text (Weight:1) \+ chart (Weight:3) \= 合計 Weight: 4 となる。  
* **Grid割り当てへの影響**: Weight: 4 という非常に重い要素と判断され、窮屈な3カラムには配置せず、広々とした TwoColumnSplitGrid の半分、または SingleColumnGrid に配置する。

### **2.3 描画とPPTXエクスポートにおける「座標のバケツリレー」**

ネスト構造を破綻させず、かつ pptxgenjs で確実に絶対座標出力するためのメカニズム。

親Gridから渡された領域（Zone）を、Cardが自身の余白を引いて内寸を計算し、中のChartにさらに小さなZoneとして引き継ぐ（相対座標の連鎖）。

## **3\. 高次ブロック（特殊図解・ナビゲーション）とフォールバック戦略**

ビジネスで頻出する「特殊な図解」や「構造的ナビゲーション」に対するアーキテクチャ方針。

LLMに直接描画（SVG生成等）を任せるアプローチは、「PPTX出力時に編集不可になる」「レイアウト崩壊」の欠陥があるため採用せず、「汎用的な高次ブロック（High-Order Blocks）の事前定義」を主軸とする。

### **3.1 高次ブロック（High-Order Blocks）の定義**

頻出パターンを専用の type としてJSONスキーマ化し、React側でテーマに沿って美しくレンダリングする。PPTX出力時は pptxgenjs のネイティブ図形（Shape/Text）に変換するため、**PowerPoint上で100%再編集可能**となる。

想定される高次ブロックの定義：

1. **agenda (アジェンダ / 目次) 【重要】**  
   * 単なる箇条書き(list)ではなく、プレゼンテーションの進行状況を示す専用ナビゲーションブロック。  
   * **機能**: active\_index プロパティを持ち、現在話している章（中間アジェンダ）をテーマに合わせて自動的にハイライト（他をグレーアウト等）する。また、時間配分(duration)やサブタイトル等のメタデータを構造的に保持する。  
2. **timeline / process\_flow (タイムライン / プロセス)**  
   * 時系列や手順。direction で縦横を指定。  
3. **matrix\_2x2 (2x2 マトリックス)**  
   * ポジショニングマップ。0.0〜1.0の相対座標で items をプロットする。  
4. **funnel (ファネル / ピラミッド)**  
   * コンバージョン推移や階層構造を示す台形図。  
5. **kpi\_metrics (特大数字ハイライト)**  
   * 業績ハイライト用。トレンド（上昇・下降）をメタデータとして持ち、UIで矢印や色を強調表示する。  
6. **venn\_diagram / cycle (ベン図 / サイクル図)**  
   * 円の重なりや、循環（ループ）プロセスを示す。

### **3.2 フォールバック・エスケープハッチ (mermaidブロック)**

事前に定義した高次ブロックでは表現できない「真にイレギュラーな相関図や概念図」を描画したい場合の最終手段。

* **JSONスキーマ:** type: "mermaid" とし、content にMermaid記法を格納する。  
* **処理方針:** LLMはテキスト構造であるMermaid記法を使うため論理破綻しにくい。PPTX出力時はネイティブ図形への変換が不可能なため、React側でレンダリングされたSVGをPNG化して貼り付ける（PowerPoint上での再編集不可は許容する）。

## **4\. 実装上の留意点**

* **ネストの深さ制限**: 無限のネストはUIレイアウトを破綻させるため、JSONスキーマのバリデーションレベルで「ネストの深さは最大1〜2階層まで」と制限する。  
* **フォールバックUI**: コンテナ内の領域が小さくなりすぎて描画が困難な場合、React側で「overflow: hidden \+ 拡大表示ボタン」などのUIフォールバックを用意する。
