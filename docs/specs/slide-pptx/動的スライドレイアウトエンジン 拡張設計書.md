# **動的スライドレイアウトエンジン 拡張設計書**

## **テーマ: 視線誘導、ネスト構造、および高次ブロック戦略**

本ドキュメントは、「動的スライドレイアウトエンジン 基本設計書」に対する高度な拡張仕様を定義する。

LLMの「デザインの自由度（CSS等の直接操作）」を制限しつつも、「情報の強調意図（視線誘導）」、「複雑な階層表現（ネスト）」、および「高度なビジネス図解・ナビゲーション」を完全に担保するためのアーキテクチャである。

## **1\. セマンティック視線誘導 (Semantic Direction & Emphasis)**

LLMが「どう見せたいか（スタイル）」ではなく、「何を見せたいか（意味・意図）」を出力し、それをUIのテーマエンジンが最適な装飾に翻訳するシステム。

### **1.1 視覚的強調（Emphasis / Highlight）**

要素をハイライトさせ、ユーザーの視線を釘付けにするためのプロパティ。

* **JSONスキーマ定義:**  
  各ブロック、またはブロック内の特定の要素に対して emphasis プロパティ（列挙型）を付与できる。  
  {  
    "type": "comparison\_table",  
    "emphasis\_row": 1, // データの2行目を強調  
    "rows": \[  
      \["A案", "低コスト", "時間がかかる"\],  
      \["B案", "高コスト", "即効性がある"\] // この行が強調される  
    \]  
  }

* **テーマごとの翻訳ロジック（React描画時の処理）:**  
  * theme-corporate-modern（コンサル風）の場合: 強調行の背景色を \#EBF4FA (薄い青)にし、テキストを太字にする。  
  * theme-tech-startup（ベンチャー風）の場合: 強調行の境界線に 2px solid var(--accent-neon) を付与し、Z-indexを上げて浮き出させる（Drop shadow）。

### **1.2 視線フローの制御（Direction）**

プロセスやタイムラインにおいて、ユーザーの視線をどのように誘導するかを決定するプロパティ。

* **JSONスキーマ定義:**  
  {  
    "type": "process\_flow",  
    "direction": "horizontal", // "horizontal" (左から右) または "vertical" (上から下)  
    "show\_arrows": true,  
    "steps": \["現状分析", "課題特定", "施策立案"\]  
  }

* **UI処理とGrid配置への影響:**  
  * direction: "horizontal" の場合、内部の要素間にテーマに沿った「➔」アイコンや結合線をレンダリングする。  
  * 推論エンジンはこの direction を読み取り、もし horizontal で要素数が多い場合は、横幅を広く取れる RowStackGrid の下段に自動的に配置する。

## **2\. コンテナ・ブロックとネスト構造 (Container & Nested Blocks)**

複雑なビジネス資料（例：カードの中にグラフを配置する、左右分割の片方にさらにリストを配置する）を表現するため、「自身の中に blocks 配列を持てるブロック（コンテナ）」を導入する。

### **2.1 コンテナ・ブロックのスキーマ定義**

特定の type（例: card, column\_group）は、再帰的に blocks プロパティを持つことができる。

{  
  "type": "card",  
  "title": "A社のパフォーマンス",  
  "emphasis": "high",  
  "blocks": \[  
    {  
      "type": "text",  
      "content": "売上高は前年比150%を達成し、業界トップの成長率を記録。"  
    },  
    {  
      "type": "chart",  
      "chart\_type": "bar",  
      "data": \[/\* ... \*/\]  
    }  
  \]  
}

### **2.2 推論エンジンにおける「再帰的重み計算」**

ネストを許容した場合、推論エンジンは「親コンテナの重み」を正確に計算しなければならない。

* **基本ルール**: コンテナの重み \= コンテナ自身の基本重み \+ 内部(blocks)の重みの合計。  
* **計算例**: 上記JSONの card の場合、text (Weight:1) \+ chart (Weight:3) \= 合計 Weight: 4 となる。  
* **Grid割り当てへの影響**: Weight: 4 という非常に重い要素と判断され、窮屈な3カラムには配置せず、広々とした TwoColumnSplitGrid の半分、または SingleColumnGrid に配置する。

### **2.3 描画とPPTXエクスポートにおける「座標のバケツリレー」**

ネスト構造を破綻させず、かつ pptxgenjs で確実に絶対座標出力するためのメカニズム。

親Gridから渡された領域（Zone）を、Cardが自身の余白を引いて内寸を計算し、中のChartにさらに小さなZoneとして引き継ぐ（相対座標の連鎖）。

### **2.4 ネスト空間における文脈認識型タイポグラフィ (Context-Aware Typography)**

コンテナ内に要素（テキストやグラフ等）がネストされた場合、限られた空間での情報密度を最適化するため、システムが自動的にテキストの階級（Variant）を降格・調整する機能。

*   **課題**: 全画面用の本文（Body）サイズをそのままカード内に適用すると、テキストが巨大化してしまい、主役であるはずのグラフや数値を圧迫してしまう。
*   **解決策**: `SlideText` は自身がコンテナ内（`isInsideContainer=true`）に配置されていることを検知すると、デフォルトの階級を `body` から `caption`（解説文サイズ）へと自動フォールバックさせる。
*   **効果**: LLM側が「このテキストはカードの中だから小さくして」と明示的なスタイル指定を行わなくても、フロントエンド側が「グラフは大きく、説明文は邪魔にならないように小さく」という視覚的黄金比を自律的に担保する。

## **3\. 高次ブロック（特殊図解・ナビゲーション）とフォールバック戦略**

ビジネスで頻出する「特殊な図解」や「構造的ナビゲーション」に対するアーキテクチャ方針。

LLMに直接描画（SVG生成等）を任せるアプローチは、「PPTX出力時に編集不可になる」「レイアウト崩壊」の欠陥があるため採用せず、「汎用的な高次ブロック（High-Order Blocks）の事前定義」を主軸とする。

### **3.1 高次ブロック（High-Order Blocks）の定義**

頻出パターンを専用の type としてJSONスキーマ化し、React側でテーマに沿って美しくレンダリングする。PPTX出力時は pptxgenjs のネイティブ図形（Shape/Text）に変換するため、**PowerPoint上で100%再編集可能**となる。

想定される高次ブロックの定義：

1. **agenda (アジェンダ / 目次) 【重要】**  
   * 単なる箇条書き(list)ではなく、プレゼンテーションの進行状況を示す専用ナビゲーションブロック。  
   * **機能**: active\_index プロパティを持ち、現在話している章（中間アジェンダ）をテーマに合わせて自動的にハイライト（他をグレーアウト等）する。また、時間配分(duration)やサブタイトル等のメタデータを構造的に保持する。  
   * **非ハイライト時の取り扱い**: 表紙の次のスライドなど、プレゼンテーション全体の目次として機能させる（特定の章をハイライトする必要がない）場合は、`active_index: -1` またはプロパティ自体を省略する。このとき、システムは「非ハイライトモード」として動作し、すべての項目を減衰（グレーアウト）させることなく、均等に高いコントラストと通常のテキストスタイルで表示して、視認性を最大化する。
2. **timeline / process\_flow (タイムライン / プロセス)**  
   * 時系列や手順。direction で縦横を指定。  
3. **matrix\_2x2 (2x2 マトリックス)**  
   * ポジショニングマップ。0.0〜1.0の相対座標で items をプロットする。  
4. **funnel (ファネル / ピラミッド)**  
   * コンバージョン推移や階層構造を示す台形図。  
5. **kpi\_metrics (特大数字ハイライト)**  
   * 業績ハイライト用。トレンド（上昇・下降）をメタデータとして持ち、UIで矢印や色を強調表示する。  
6. **venn\_diagram / cycle (ベン図 / サイクル図)**  
   * 円の重なりや、循環（ループ）プロセスを示す。

### **3.2 フォールバック・エスケープハッチ (mermaidブロック)**

事前に定義した高次ブロックでは表現できない「真にイレギュラーな相関図や概念図」を描画したい場合の最終手段。

* **JSONスキーマ:** type: "mermaid" とし、content にMermaid記法を格納する。  
* **処理方針:** LLMはテキスト構造であるMermaid記法を使うため論理破綻しにくい。PPTX出力時はネイティブ図形への変換が不可能なため、React側でレンダリングされたSVGをPNG化して貼り付ける（PowerPoint上での再編集不可は許容する）。

### **3.3 表ブロック（comparison\_table）の高度な強調仕様**

表ブロック内の一部情報に注意を引きつけるための視線誘導メカニズムとして、従来の行単位の強調（`emphasis_row`）に加え、セル単位での細分化された強調およびグルーピング強調を定義する。

#### **3.3.1 個別セル強調 (`emphasis_cells`)**
* **スキーマ**: `emphasis_cells: [number, number][]` (行および列の 0-indexed 座標の二次元配列)
* **描画仕様**: 指定されたセルのフォントを太字（`font-weight: bold`）にし、背景色を極薄いプライマリカラー（`var(--bg-table-highlight-cell, rgba(0, 32, 91, 0.05))`）に塗りつぶす。

#### **3.3.2 グルーピング強調（コールアウトボックス `emphasis_box`）**
複数の隣接するセルを1つの太い枠線（コールアウト）で囲むための仕様。
* **スキーマ**:
  ```json
  "emphasis_box": {
    "start_row": number,
    "end_row": number,
    "start_col": number,
    "end_col": number
  }
  ```
* **描画仕様（外周境界線制御方式）**:
  画面レスポンシブなサイズ伸縮に追従させるため、絶対配置の重ね合わせを避け、対象範囲 `[start_row 〜 end_row, start_col 〜 end_col]` のセル境界線を個別に計算して太い境界線を適用する。
  * `rowIdx === start_row` (上端セル) ➡️ 上境界線を太く (`border-top: 3px solid var(--accent-color, #e11d48)`)
  * `rowIdx === end_row` (下端セル) ➡️ 下境界線を太く (`border-bottom: 3px solid var(--accent-color, #e11d48)`)
  * `colIdx === start_col` (左端セル) ➡️ 左境界線を太く (`border-left: 3px solid var(--accent-color, #e11d48)`)
  * `colIdx === end_col` (右端セル) ➡️ 右境界線を太く (`border-right: 3px solid var(--accent-color, #e11d48)`)
  * **範囲内共通**: 背景を薄い警告・アクセント色（`var(--bg-table-highlight-box, rgba(225, 29, 72, 0.04))`）でハイライト。

### **3.4 画像プレースホルダーブロック（image\_placeholder）の仕様**

#### **3.4.1 背景と導入意図 (Background & Rationale)**
従来のマークアップブロック（テキスト、リスト、テーブル、単純なグラフ、Mermaid等）では、現実のビジネス資料に不可欠な「複雑な業務フロー」「詳細なシステム構成図」「推進体制の組織ポンチ絵」「画面のモックアップ」などをLLMがリアルタイムに高品位なコード（React/CSS/SVG）として出力するのは極めて困難であり、ハルシネーションや描画崩れを引き起こす原因となっていた。

本ブロックは、スライド生成時に「ここにどのようなビジュアルが配置されるべきか」という論理的な情報構造（文脈、生成プロンプト、検索クエリ等）をプレースホルダー（仮置き枠）として保持・描画するエスケープハッチである。これにより、プレゼンテーション全体のストーリーや構造を維持したまま、将来的に外部の画像生成API（DALL-E 3等）や画像アセット取得ライブラリと結合して実画像に自動置換できる仕組みを提供する。

#### **3.4.2 データ構造（スキーマ定義）**
JSONドキュメントにおける `image_placeholder` ブロックのオブジェクト構造を以下のように定義する。

| プロパティ名 | 型 | 必須 | 役割・説明 |
| :--- | :--- | :--- | :--- |
| `type` | `string` | **必須** | 固定値 `"image_placeholder"`。 |
| `label` | `string` | **必須** | プレースホルダー中央に描画されるビジュアル挿入の指示ラベル（例：「DX推進体制の組織図」）。 |
| `prompt` | `string` | 任意 | 画像生成API（DALL-E 3等）に送るための英語のグラフィック生成用詳細プロンプト。 |
| `search_query` | `string` | 任意 | ストックフォトAPIや画像検索APIに送るための、ビジネス文脈に沿った検索キーワード。 |
| `aspect_ratio` | `string` | 任意 | 確保するアスペクト比。 `"16:9"` / `"4:3"` / `"1:1"` から選択。未指定時は `"16:9"`。 |
| `image_url` | `string` | 任意 | 生成または検索によって取得が完了した実画像のURL。値がある場合はプレースホルダー表現に代わり実画像を領域に敷き詰めて表示。 |
| `fallback_text` | `string` | 任意 | 画像が読み込めなかった、または支援技術向けの代替説明テキスト。 |

#### **3.4.3 描画・UI仕様**
* **ワイヤーフレーム表現 (仮置き状態)**:
  `image_url` が空、または定義されていない場合、デザイナーのモックアップ作成時と同様のワイヤーフレーム表現を行う。外周を薄い破線（`dashed border`）で囲み、領域内に対角のクロスライン（SVG）をうっすらと描画する。
* **API連携用デバッグメタデータトグル**:
  プレースホルダー内に「生成API設定を表示 (Prompt / Query)」というトグルボタンを配備し、クリックすると `prompt` および `search_query` の中身を整形したコードブロック（monospaceフォント）で展開表示する。これにより、将来のAPI統合に向けたプロンプトエンジニアリングのデバッグをスライド上で行うことができる。
* **アスペクト比のレスポンシブ維持**:
  コンテナサイズ（`cqi`）が縮小・拡大しても、指定された `aspect_ratio` の縦横比率をCSS（`aspect-ratio` プロパティ）を用いて厳密に維持し、グリッド全体のレイアウト崩れを防ぐ。

## **4\. 実装上の留意点**

* **ネストの深さ制限**: 無限のネストはUIレイアウトを破綻させるため、JSONスキーマのバリデーションレベルで「ネストの深さは最大1〜2階層まで」と制限する。  
* **フォールバックUI**: コンテナ内の領域が小さくなりすぎて描画が困難な場合、React側で「overflow: hidden \+ 拡大表示ボタン」などのUIフォールバックを用意する。
