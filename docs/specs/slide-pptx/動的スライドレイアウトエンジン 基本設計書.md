# **動的スライドレイアウトエンジン (Dynamic Slide Engine) 基本設計書**

## **1\. コンセプトと目的**

既存の「目的特化型・固定レイアウト（ChartSlide等）」から脱却し、「コンテンツ駆動型・自動レイアウト」へとアーキテクチャを転換する。

LLMにはスライドの「論理的構造（意味的ブロック）」のみを生成させ、フロントエンドの推論エンジンが最適な「グリッド空間（レイアウトの型）」を自動選択して描画する。

これにより、以下の3点を同時に実現する。

1. **情報伝達の最適化**: マッキンゼー流の「ワンスライド・ワンメッセージ」と「情報の構造化」を強制。  
2. **デザインの柔軟性**: 多様なブロック（図表、リスト、プロセス図）を美しく整列・配置。  
3. **PPTX出力の安定性**: 動的でありながら論理的なグリッド構造に落とし込むことで、pptxgenjs の座標計算を確実なものにする。

## **2\. 概念の転換：目的型から構造型へ**

本アーキテクチャの最大のポイントは、テンプレートの概念を「コンテンツの種類」から「空間の分割構造」へと転換した点にある。

### **2.1 従来のアプローチ（目的型・固定テンプレート）**

* **例**: TableSlide (表用スライド), ChartSlide (グラフ用スライド)  
* **概念**: 「ここに表を入れる」「ここにグラフを入れる」という、**目的ベース**の専用枠。  
* **課題**: 柔軟性に乏しい。「表とグラフを横に並べたい」という要求が発生するたびに、新しい専用テンプレート（例: TableAndChartSlide）を開発しなければならず、表現の幅に限界があった。

### **2.2 新しいアプローチ（構造型・動的レイアウト）**

* **例**: SingleColumnGrid (1カラム), TwoColumnSplitGrid (左右分割)  
* **概念**: コンテンツの中身（表かグラフか）には一切依存しない、**純粋な空間の分割パターン（グリッド）**。  
* **メカニズム**: フロントエンドの推論エンジンが、LLMから渡された「ブロック群（情報）」を解析し、「テトリスのように、最も美しく収まる箱（グリッド）を自動的に選択」して当てはめる。  
* **メリット**: コンテンツ（意味）とレイアウト（箱）が完全に分離されるため、未知の組み合わせであってもシステムが自動で最適化し、プロの手による資料のように厳格な整列（アラインメント）を保ったまま柔軟な描画が可能となる。

## **3\. システム・アーキテクチャ概要**

システムは以下の4つのレイヤーで構成される。

### **Layer 1: 論理データ層 (JSON Schema)**

LLMが出力するデータ。レイアウトに関する指示（「左に配置」「50%の幅」など）は一切持たせず、純粋な情報構造のみを定義する。

* **必須要件**: 全てのスライドに key\_message（リード文）を義務付ける。  
* **構成要素**: 情報は blocks 配列に格納される。

### **Layer 2: レイアウト推論エンジン (Inference Engine)**

Reactフロントエンドにおける頭脳。Layer 1で受け取った blocks の「要素数」「種類」「重み（視覚的ボリューム）」を解析し、Layer 3で用意されたどのグリッド・コンポーネントに流し込むかを決定する。

### **Layer 3: グリッド＆ブロック描画層 (React Components)**

* **Grid Components**: 推論エンジンによって選ばれる「空間の型」。要素を美しく整列（アラインメント）させるためのコンテナ。  
* **Block Components**: json\_document の資産を流用・拡張した、各情報要素（テキスト、表、Mermaid等）の描画コンポーネント。

### **Layer 4: エクスポート層 (PPTX Export Engine)**

Reactでの描画構造（Grid \-\> Block）と1:1で対応する pptxgenjs の生成ロジック。Gridが持つ座標情報（X, Y, W, H）を基準に、各BlockをPPTX上に配置する。

## **4\. データスキーマ定義案 (Layer 1\)**

LLMには以下のフォーマットで出力させる。key\_message はスライドの結論を示す最重要項目である。

{  
  "slides": \[  
    {  
      "key\_message": "メンタリングの質向上のため、アルゴリズムではなくメンターがメンティーを選定する方式を採用した。",  
      "blocks": \[  
        {  
          "type": "text",  
          "content": "従来の手法と新しいアプローチの比較は以下の通りです。"  
        },  
        {  
          "type": "comparison\_table",  
          "headers": \["アプローチ", "Pros", "Cons"\],  
          "rows": \[  
            \["自動マッチング", "工数が少ない", "細かい調整が必要"\],  
            \["希望ベース選定", "オーナーシップ向上", "選定に時間がかかる"\]  
          \]  
        }  
      \],  
      "speaker\_notes": "このスライドではマッチング手法の変更理由を説明します..."  
    }  
  \]  
}

## **5\. レイアウト推論エンジンの仕様 (Layer 2 & 3\)**

推論エンジンは、ブロックの配列を解析し、最適なグリッドを適用する。この判断基準となるのが「重み（Weight）」の概念である。

### **5.1 ブロックの「重み（Weight）」の定義**

各ブロックタイプには、視覚的なボリュームや複雑さを示す「重み」スコアを事前に定義しておく。これにより、エンジンは情報の密度を計算できる。

* **Weight 1 (軽量)**: text (短い段落), list (シンプルな箇条書き)  
* **Weight 2 (中量)**: key\_value\_card (アイコン付きカード), process\_step (シンプルな手順)  
* **Weight 3 (重量)**: comparison\_table (表), chart (グラフ), mermaid (複雑な図解), image\_placeholder (画像・図表プレースホルダー)

### **5.2 基本グリッド（空間の型）と適応条件**

定義された重みと要素数に基づき、以下のいずれかのグリッドに割り当てる。

1. **SingleColumnGrid (1カラム)**  
   * **用途**: 画面全体を使って1つの情報を大きく見せる。  
   * **適応条件**: 要素数が1つ、または「重み3」の要素が1つだけ存在する場合。  
2. **TwoColumnSplitGrid (左右分割)**  
   * **用途**: テキスト（説明）と図表（根拠）の黄金比レイアウト。コンサルティングスライドの標準に則り、左列にメインメッセージ（最初の要素）、右列に図表・カード群（2つ目以降の全要素）を強制配置する。  
   * **適応条件**: 要素数が2つ、または4つ以上の場合、あるいは「重み3（Heavy）」の要素が含まれる場合に最優先で適用され、縦方向のコンテンツ溢れ（オーバーフロー）を強力に防止する。  
3. **ThreeColumnMultiGrid (3カラム)**  
   * **用途**: 3つの特徴、3つのプロセスなどを横に並べて比較・強調する。  
   * **適応条件**: 要素数が3つで、すべてが「重み1〜2」の同種の要素（例：key\_value\_card × 3）である場合。  
4. **RowStackGrid (水平分割・上下)**  
   * **用途**: 上段に説明、下段に横長の要素（タイムラインやプロセスフローなど）を配置。  
   * **適応条件**: 要素数が2つで、下部の要素が横幅を必要とするタイプ（横長のプロセス図など）の場合。

### **5.3 主要な独自ブロックコンポーネント (Block Components)**

本エンジンの構築に際し、高密度な情報レイアウトとデザイン性を両立させるために、以下の専用ブロックコンポーネントを新規実装・拡張した。

*   **SlideText (`text`) / SlideList (`list`)**:
    *   **用途**: 本文、解説文、および箇条書き。
    *   **特徴**: 物理的なサイズ（px, cqi）の直接指定を廃止し、「セマンティック・タイポグラフィ・システム」を導入。情報が `KeyMessage`、`CardTitle`、`Caption` 等の役割（Variant）を持ち、それに応じて自動的に適切なサイズ・行間でレンダリングされる。
*   **SlideCard (`key_value_card`)**:
    *   **用途**: KPIなどの重要な「タイトルと単一の数値・指標」を強調表示する。
    *   **特徴**: コンサルティングテーマでは左側にアクセント線が入り、数値を大きくシャープに表現する。
*   **SlideContentCard (`content_card`)**:
    *   **用途**: タイトル、説明文、さらに補足の箇条書き（points）を一つの枠組みとしてグループ化する。
    *   **特徴**: 情報の塊（チャンク）を視覚的に分断させないため、上部にブランドカラーのヘッダーを持つカードデザイン。
*   **SlideChart (`chart`)**:
    *   **用途**: 定量データのダイナミックな可視化。
    *   **サポートグラフ**:
        *   `bar` (横棒グラフ): 項目の比較。
        *   `column` (縦棒グラフ): 時系列推移やカテゴリ比較。
        *   `line` (折れ線グラフ): トレンドの可視化。
        *   `donut` (ドーナツグラフ): 構成比・シェアの可視化。
    *   **特徴**: SVGやCSSの `linear-gradient` と動的な幅計算を用いて、リッチな質感とアニメーションを持つグラフを表現。主役としての視覚的ウェイト（高めのcqi）を確保しつつ、PPTX出力時にはネイティブグラフに自動変換される。
*   **SlideMermaid (`mermaid`)**:
    *   **用途**: アーキテクチャ図、システム構成、フローチャートなどの複雑な構造図。
    *   **特徴**: CSS Flexboxの高度な収縮計算（`minHeight: 0`）と SVGのプロポーショナル縮小設定を組み合わせることで、どんなに巨大な図でも親グリッドを絶対に溢れさせず、自動的に枠内にスケールダウンさせる「オーバーフロー保護機構」を持つ。
*   **SlideImagePlaceholder (`image_placeholder`)**:
    *   **用途**: 標準のブロック（テキスト、表、グラフ等）では表現が極めて難しい「複雑な組織図、概念図、画面のポンチ絵」などを配置するスペースを確保するための仮置きブロック（プレースホルダー）。
    *   **特徴**: `image_url` が未指定のときは、設計図（ワイヤーフレーム）を模した破線枠と対角クロス線（SVG）で領域を描画。将来の画像生成API（DALL-E等）やストックフォトAPIとの統合を視野に入れ、生成用のプロンプトや検索キーワードを展開・確認できるアコーディオンUIを内蔵。アスペクト比指定に対応し、レイアウト計算時にも比率を維持する。

## **6\. テーマ管理アーキテクチャ (Theming System)**

新しいアーキテクチャでは、「コンポーネントの複製（テーマごとのファイル作成）」を廃止し、**「デザイントークン」と「CSS変数」による一元的なテーマ注入方式**を採用する。

これにより、コアとなる Grid や Block のロジックは全テーマで100%共有される。

### **6.1 テーマの構成要素**

テーマ（例：modern-indigo）は、以下の3つの要素のみを提供する。

1. **デザイントークン (CSS Variables)**  
   * theme.css にて、--primary-color, \--bg-color, \--font-heading, \--spacing-base などの変数を定義。  
   * ユーザーがテーマを切り替えると、スライドのルート要素（例：\<div class="theme-modern-indigo"\>）が切り替わり、すべてのブロックの色やフォントが一瞬で変更される。  

### **6.2 セマンティック・タイポグラフィ・システム (Typography Scale)**

旧来の「要素に直接 `font-size: 1.6cqi` や `13px` をハードコードする」方式から脱却し、スライド内の全てのテキストをCSS変数（タイポグラフィ・トークン）で一元管理する仕組み。

#### **6.2.1 タイポグラフィ・トークン一覧（既定値）**

スライドの横幅（コンテナサイズ）に応じて自動的に拡大縮小するよう、単位には `cqi` (Container Query Inline-size) を使用する。

| CSS変数名 | フォントサイズ | 行間 (`line-height`) | 役割・主な用途 |
| :--- | :--- | :--- | :--- |
| `--text-cover-title` | `4.2cqi` | `1.3` | 表紙のメインタイトル用 |
| `--text-cover-subtitle` | `1.8cqi` | `1.3` | 表紙のサブタイトル用 |
| `--text-key-message` | `2.4cqi` | `1.3` | スライド最上部のキーメッセージ（結論文） |
| `--text-title` | `2.0cqi` | `1.3` | スライドタイトル / 大見出し |
| `--text-section-header` | `1.7cqi` | `1.3` | セクションの見出し / 中見出し |
| `--text-card-title` | `1.5cqi` | `1.3` | KPIカードのタイトル / グループ小見出し |
| `--text-body-large` | `1.4cqi` | `1.6` | 箇条書きリストなどの、少し大きめの本文 |
| `--text-body` | `1.2cqi` | `1.6` | スライド内の標準本文テキスト |
| `--text-caption` | `1.1cqi` | `1.4` | 図表の解説テキスト / ネストされた空間の補足文 |
| `--text-label` | `0.85cqi` | `1.4` | グラフの軸ラベル / 欄外の極小注記 |

#### **6.2.2 堅牢なテーマ上書き規則とカスタマイズ例**

テーマ側（例：コンサルクラシックテーマ）で文字サイズを調整したい場合、コンポーネントのクラスに対して直接 `font-size: 13px !important;` を指定することは禁止する。代わりに、テーマのルートクラス配下でCSS変数の値のみをオーバーライドする。

*   **コンサルクラシックテーマでの変形例（引き締め）**:
    ```css
    .theme-consulting-classic {
        --text-body: 1.15cqi;     /* 1.2cqi から若干縮小して高密度化 */
        --text-caption: 1.05cqi;  /* 1.1cqi から縮小してグラフと解説のメリハリを強化 */
    }
    ```
    このように定義することで、画面のレスポンシブ拡大縮小性能（Container Query）を完全に維持したまま、テーマ独自のデザイン比率を注入可能となる。
2. **ブロック・スタイリング (Block Styles)**  
   * ブロックコンポーネント（例：comparison\_table）の実体は1つだが、テーマごとに「表のヘッダーを塗りつぶすか」「罫線のみにするか」といったスタイルバリエーションを適用する。  
   * クラス名の工夫（例：.theme-modern-indigo .doc-table { ... }）により、コアのReactコードを汚さずに見た目を制御する。  
3. **装飾レイヤー (Decorations)**  
   * テーマごとに特有のあしらい（背景の幾何学模様、タイトルの下のラインなど）。  
   * SlideRenderer 側でテーマ設定を読み込み、共通の Grid コンポーネントの背面に「テーマ固有の装飾コンポーネント」を透過レイヤーとして敷く。

### **6.2 PPTX出力時のテーマ適用**

themeRegistry.js にて、テーマごとの「カラーパレット（Hex値）」や「フォント名」を定義しておく。pptxExportEngine は出力時に現在アクティブなテーマの設定値を読み込み、pptxgenjs の描画オブジェクト（ShapeやText）に対して動的に色やフォントを注入する。

### **6.3 表現の幅（ユースケース例）**

構造（Grid）と装飾（Theme）が分離されているため、LLMが同じ情報（JSON）を出力しても、テーマを切り替えるだけで全く異なる印象を与えることができる。

* **UseCase A: 戦略コンサル・スタイル (例: theme-consulting-classic)**  
  * **Tokens**: 白背景、濃紺のアクセント、明朝体またはフォーマルなゴシック体。  
  * **Blocks**: 装飾を削ぎ落とし、細い実線のみで構成された表や図。余白は計算され尽くした均等配置。  
  * **Decorations**: なし（ノイズの排除）。ロジックと整列の美しさを強調。  
* **UseCase B: メガベンチャー・ピッチスタイル (例: theme-tech-startup)**  
  * **Tokens**: ダークモードまたは鮮やかなグラデーション背景、極太のモダンなサンセリフ体（巨大なフォントサイズ）。  
  * **Blocks**: 背景を少し透過させたダークパネル、強めのドロップシャドウ、角丸（border-radius）を多用。テキスト色は白やネオンカラー。  
  * **Decorations**: 背景にブランドカラーのフレアグラデーションや、幾何学的なメッシュレイヤーを配置。エモーショナルでストーリー性の強いインパクトを強調。

## **7\. PPTX エクスポートとの連携 (Layer 4\)**

このアーキテクチャの最大の利点は、**「グリッド」がPPTX上の「座標（x, y, w, h）」を絶対的に保証する**ことにある。

1. **グリッドによる領域（Zone）の定義**:  
   例えば TwoColumnSplitGrid は、内部的に明確な領域情報を持つ。  
   * LeftZone \= { x: '5%', y: '20%', w: '40%', h: '70%' }  
   * RightZone \= { x: '50%', y: '20%', w: '45%', h: '70%' }  
2. **ブロックへの座標制約の伝播**:  
   PPTXエクスポート処理時、推論エンジンが決定したグリッドの「領域情報（Zone）」が、子要素である各ブロックに引き渡される。ブロック（例：comparison\_table）は、**「自分がどのZoneに配置されたか」を知った上で、その枠（W, H）に収まるように**セルの幅やフォントサイズを計算して pptxgenjs のAPIを叩く。  
3. **ネイティブ描画とSVGベクター直接埋め込み (Native & Vector Graphics)**:  
   chartブロックはPptxGenJSの `addChart()` を用いて、PowerPoint上で後から数値を直接再編集可能な「本物のグラフ」としてネイティブ出力する。また、mermaid等の複雑な図解ブロックについては、DOMの画面キャプチャ（html2canvas等）の不安定な手法には頼らず、出力プロセス内でバックグラウンド実行（`mermaid.render()`）を行い、生成されたSVGデータをBase64エンコードして「ベクター画像」としてPPTXに直接埋め込む。これにより、いかなる解像度でも画質劣化のないプロフェッショナルな出力を保証する。

## **8\. 実装フェーズ（マイルストーン）**

### **Phase 1: コアエンジンの構築**

* 新JSONスキーマの定義とLLMプロンプト（システムプロンプト）の設計。  
* 重み付け（Weight）のルール定義。  
* SlideRenderer.jsx の刷新（推論エンジンの実装とロジックの組み込み）。  
* 空のGridコンポーネント（Single, Split, Multi, Stack）の作成。

### **Phase 2: ブロックの移植とテーマシステムの統合**

* json\_document から基本的なブロックを json\_slide 向けに移植。  
* themeRegistry.js の改修。CSS変数を用いたグローバルなテーマ注入システムの構築。  
* テーマ固有のコンポーネント（themes/modern-indigo/slides/ 配下のファイル）の統廃合・削除。

### **Phase 3: PPTXエクスポーターの再構築**

* pptxExportEngine.ts を改修し、GridのZone（領域）ベースの座標マッピングと、テーマカラーの動的注入ロジックを実装。  
* 複雑なブロックの画像化と、Zoneへの埋め込み処理の実装。
