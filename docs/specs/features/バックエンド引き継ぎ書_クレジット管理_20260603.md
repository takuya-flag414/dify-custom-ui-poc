## 1. 目的と現状のステータス

フロントエンド側にて、AIエージェントのクレジット（利用可能枠）管理機能のUIおよびモックAPIによるロジック分離が完了しました。
本ドキュメントは、モックとして稼働している機能を本番環境（Firebase Cloud Functions / Firestore）へ移行するための実装要件をまとめたものです。

**フロントエンド側の完了事項:**
- サイドバーへの残高表示、枯渇時の入力ロック機構、消費クレジットのバッジ表示UIの実装。
- ビジネスロジック（残高計算、リセット日の算出など）をすべてUIから切り離し、`src/api/creditApi.ts` というAPIラッパー層に押し出しました。
- これにより、本番移行時は **「フロントエンド側のコードには一切触れず、APIラッパー層の関数をFirebase呼び出しに差し替えるだけ」** で完了する状態になっています。

## 2. 実装をお願いしたいエンドポイント (Cloud Functions)

フロントエンドが現在依存している4つのAPIエンドポイントの実装をお願いします。
（フロントエンドの `src/api/creditApi.ts` の内容を実際の関数に置き換える想定です）

### ① `fetchUserCredit(userId: string)`
*   **概要:** 該当ユーザーの現在のクレジット残高と、次回リセット予定日を取得します。
*   **戻り値:** `{ balance: number, nextResetDateStr: string }`
    *   `nextResetDateStr` は "〇月〇日" のようなフロント表示用の文字列としてバックエンド側で計算して返してください。（フロントのUIを極力ダム（受動的）に保つため）

### ② `deductUserCredit(userId: string, amount: number)`
*   **概要:** メッセージ送信完了時（またはストリーミング終了時）に、消費したクレジットを減算します。
*   **備考:** Dify側の `total_price` や `total_cost` の集計をバックエンド側（プロキシ層）で行う設計案もありますが、現在のフロントエンドは「使った分をAPIに申告して減らしてもらう」設計になっています。本番ではプロキシ層での完全自動引き落としにするかご相談させてください。

### ③ `addUserCredit(userId: string, amount: number)`
*   **概要:** クレジットを通常加算します（現状未使用ですが拡張用です）。

### ④ `grantBonusCredit(adminUserId: string, targetUserId: string, amount: number)`
*   **概要:** 【管理者用】例外的に枯渇してしまったユーザーに対し、管理者がボーナス枠を手動で付与します。
*   **備考:** この処理を叩けるのは `admin` ロールを持つユーザーのみとするよう、Functions側で権限チェックをお願いします。

## 3. Firestore スキーマの拡張要件

既存の `users` コレクションに対し、以下のフィールドの追加をお願いします。

| フィールド名 | 型 | 説明 | デフォルト値 |
| :--- | :--- | :--- | :--- |
| `credit_balance` | Number | 現在の残りクレジット (CR) | `10000`（※DBから取得） |
| `credit_limit` | Number | 毎週の付与上限 | `10000`（※DBから取得） |
| `last_reset_time` | Number | 最後に週次リセットが実行された日時（エポックミリ秒） | ユーザー作成時のタイムスタンプ |

### （追加）システム全体設定（Global Config）のDB化要件
プログラム内に `DEFAULT_LIMIT = 10000` のように上限値をハードコードせず、別途 `system_settings` コレクション等を作成し、そこにデフォルト上限値を持たせてください。
*   **目的:** 今後、非エンジニアの管理者でも管理画面からサクッと全ユーザーのデフォルト上限を変更（チューニング）できるようにするためです。
*   **API側の動き:** 新規ユーザーの作成時や、週次のリセット適用時に、この `system_settings` から現在のデフォルト値（例: スタンダードなら10000）を都度読み込んで付与する設計にしてください。
*   **その他ハードコードを避けるべき項目:** 上限値だけでなく、AIコストの換算レート（`usd_to_credit_rate` = 10000）、課金対象ノードの指定リスト（`llm`, `http-request`, `tool`）、価格が出力されないRAG等の固定価格表なども同様に `system_settings` から読み込み、プログラム内への直書きを避けるように設計をお願いします。

## 4. 週次リセット機能（クーロン処理）の実装方針

「毎週月曜日に全ユーザーのクレジットを上限(`credit_limit`)まで回復させる」仕様について、フロントエンドのモック環境では現在 **「アクセス時遅延評価（疑似クーロン）」** という手法で実装しています。

**モックでの実装方法:**
ユーザーが `fetchUserCredit` などのAPIを叩いた際、バックエンド側で現在時刻と `last_reset_time` を比較し、「前回のアクセスから新しい月曜日を跨いでいるか」を判定します。跨いでいれば、残高をリセットして `last_reset_time` を更新した上でレスポンスを返します。

**本番環境での選択肢:**
村上さんにて、以下のどちらの実装を採用するかご検討をお願いいたします。
1.  **アクセス時遅延評価（モック踏襲案）:** 定期実行ジョブのコストがかからず、シンプルです。
2.  **Cloud Scheduler（クーロンジョブ案）:** 毎週月曜0時にFirestoreの全ユーザーデータをバッチ更新します。全ユーザーのデータを一斉更新するため確実ですが、ユーザー数が増えた際のRead/Writeコストに注意が必要です。
