# カスタムボット機能 バックエンド開発引き継ぎ書

## 1. はじめに（引き継ぎの背景と目的）

本ドキュメントは、カスタムボット機能の開発において、藤井が担当した「Phase 1: フロントエンドモック構築と仕様・UI確定」が完了したことに伴い、バックエンド開発を担当する村上さんへ仕様を引き継ぐための資料です。
「カスタムボット機能（Custom Bots） 詳細設計および実装計画書」をベースにしつつ、フロントエンド実装の中で追加・調整された最新の仕様（ハイブリッド設定、Dify通信プロトコルの詳細、モックで実現したUI/UXなど）をまとめています。

本ドキュメントに基づき、Phase 2 として Firebase (Firestore / Auth) の基盤構築、および Cloud Functions を用いた Dify へのファイルストリーム転送APIの実装をお願いいたします。

## 2. データモデル仕様 (Firestore)

フロントエンドのモック実装で確定した、Firestoreの `custom_bots` コレクションのドキュメントスキーマは以下の通りです。

**【重要変更点】**: RAG設定（`rag_config`）とコンテキストファイル（`context_file_url`）は排他ではなく、**ハイブリッド（同時設定）可能**な設計となりました。UI上でも両方を同時に設定できる構造になっています。

```json
{
  "bot_id": "bot_xyz_789",
  "name": "営業提案書チェッカー",
  "description": "提案書の構成をチェックします",
  "system_prompt": "あなたは優秀な営業コンサルタントです...",
  "creator_uid": "user_uid_001",
  "department_id": "dept_sales",
  "visibility": "department",
  "rag_config": {
    "enabled": true,
    "target_store_id": "corpora/1234567890",
    "target_store_name": "corp_sales_rules_2026"
  },
  "context_file_url": "gs://my-app-bucket/custom_bots/bot_xyz_789/context.pdf",
  "created_at": "2026-06-11T10:00:00Z"
}
```

## 3. 認証とセキュリティルール要件

- **Firebase Auth Custom Claims**: ユーザーアカウントに所属部署を示す `department_id` を付与する仕組み（Admin SDK を使用した Cloud Functions 等）を構築してください。
- **Firestore セキュリティルール**: `visibility` の値に応じて以下の制御を行ってください。
  - `private`（非公開）: `creator_uid` が自分と一致する場合のみ Read/Write 可能。
  - `department`（部署内共有）: `department_id` が自分の所属部署と一致する場合に Read 可能。Write は作成者のみ。
  - `public`（全社公開）: 認証済みユーザーなら誰でも Read 可能。Write は作成者のみ。

## 4. Cloud Functions 実装要件（最重要課題）

**Difyへの一時ファイルストリーム転送APIの実装**が必要です。
ボットのコンテキストファイル実体が Dify でのチャット開始時に必要になりますが、Difyの一時ファイルIDはセッションごとにリセットされるため、フロントエンドからではなくバックエンド経由で「横流し」する必要があります。

### API の期待される挙動
1. **トリガー**: フロントエンドから `file_url` (`gs://...`) を受け取る（Callable FunctionsまたはREST API）。
2. **Storage読み取り**: Firebase Storage から指定されたファイルの Readable Stream を取得する。
3. **転送 (Pipe)**: メモリ上で `multipart/form-data` を構築し、Dify API (`/files/upload`) へ転送する。
4. **【必須条件】**: 転送時、Dify API の要求に従い必ず `user` フィールドに呼び出し元のFirebase Auth UIDをセットしてください。
5. **返却**: Dify から返ってきた `upload_file_id` (UUID等) をフロントエンドへ返す。

## 5. フロントエンド側の実装状況と繋ぎ込みポイント

現在、フロントエンドは `mock_custom_bots_db` 等を用いてローカルで状態管理を行っています。
バックエンド基盤の準備ができ次第、以下のファイル・関数群を本物の API に置き換える繋ぎ込み作業が発生します。

- **`src/hooks/useCustomBots.ts`**:
  フロントエンドのカスタムボット管理フック。現在、モック関数で `createBot`, `updateBot`, `fetchBots` などを処理しています。
- **Dify通信のペイロード構築 (`useChat.ts` 等)**:
  **【重要共有事項1】**: Dify側の開始ノードで受け取る変数仕様について、RAG機能が有効（`rag_config.enabled == true`）のボット利用時に `rag_enabled` が `false` になりRAG検索が発動しない不具合がありました。そのため、現在は Dify 送信時のペイロードにおいて、JSONプロトコル内だけでなく、**Difyの開始ノード変数（`inputs`）にも直接 `rag_enabled: true` を送信**するように実装されています（余計な変数は加えず、必要なものだけを送信）。
  **【重要共有事項2】Dify通信時のファイルID制限について**: 現在フロントエンドのモック動作上、カスタムボットのコンテキストファイル（StorageのURL）から一時的に仮想ファイルID（`bot_file_...`）を生成してチャット画面に認識させています。しかし、本番の Dify API は `files` に厳密な UUID 形式の `upload_file_id` を要求するため、そのまま送るとエラー（400 Bad Request）になります。そのため現状のフロントエンド（`ChatServiceAdapter.ts`）では、本番API通信時にのみこれら仮想IDをフィルタリングする暫定対応を入れています。バックエンド（Functions）実装後は、Functions が Dify の UUID を取得してフロントに返す仕様となるため、この暫定フィルタリングを解除して繋ぎ込みを行います。

## 6. 開発と結合テストのステップ

村上さんには、以下のステップで段階的にバックエンドを実装・結合していただく想定です。

1. **Step 1: 基盤構築とシステムプロンプトのみのボット実装**
   - Firestoreコレクション作成、CRUD機能の実装（モックからの置き換え）。
   - コンテキスト（RAG・ファイル）なしの状態でプロンプト制御が動くかの結合テスト。
2. **Step 2: 事前定義RAG連携機能の実装**
   - `rag_config` の保存と、フロントエンドからのDify通信でのRAGパラメータの引き渡し確認。
3. **Step 3: Functionsを利用したファイルストリーム転送機能の実装**
   - Storage保存および Functions による Dify へのアップロード（ID取得）の結合テスト。
